# 2FA-Unterstützung — Design Spec
**Datum:** 2026-05-15  
**Status:** Implementiert & getestet (Fix 2026-05-26)

---

## Übersicht

Zweistufige Authentifizierung (TOTP) beim Login. Das OTP-Feld erscheint automatisch, wenn der Server einen 2FA-Fehler zurückgibt — ohne Konfiguration, ohne UI-Rauschen für Nutzer ohne 2FA.

---

## Architektur

Zwei geänderte Dateien, keine neuen:

| Datei | Änderung |
|-------|----------|
| `Paperless TeDi/Services/PaperlessAPI.swift` | Neuer `APIError.otpRequired`, `fetchToken` mit optionalem `otp`-Parameter, OTP-Erkennung im Response |
| `Paperless TeDi/Views/Auth/LoginView.swift` | OTP-State, bedingtes OTP-Feld, angepasster Login-Flow |

---

## Flow

```
User: Server + Username + Passwort eingeben → Login tippen
    └─► fetchToken(otp: nil)
            ├─► 200 → Token speichern → App öffnen
            └─► 400/401 + OTP-Keyword → APIError.otpRequired
                    └─► otpRequired = true → OTP-Feld erscheint
                            └─► User gibt 6-stelligen Code ein → Login tippen
                                    └─► fetchToken(otp: "123456")
                                            ├─► 200 → Token speichern → App öffnen
                                            └─► 400/401 → "Falscher Code"
```

## API-Änderungen (`PaperlessAPI.swift`)

### Neuer Error-Case

```swift
enum APIError: Error, LocalizedError {
    // bestehende Cases ...
    case otpRequired
}
```

### `fetchToken` mit optionalem OTP

```swift
static func fetchToken(
    serverUrl: String,
    username: String,
    password: String,
    otp: String? = nil
) async throws -> String
```

Request-Body:
```swift
var body: [String: String] = ["username": username, "password": password]
if let otp { body["code"] = otp }  // Paperless NGX erwartet "code", nicht "otp"
```

> **Wichtig (Fix 2026-05-26):** Das Feld heißt `code`, nicht `otp`. Gefunden in
> `/usr/src/paperless/src/paperless/serialisers.py`: `code = attrs.get("code")`.

### OTP-Erkennung

Bei HTTP 400 oder 401: Response-Body als String parsen, case-insensitiv nach einem dieser Keywords suchen:
`"otp"`, `"totp"`, `"mfa"`, `"2fa"`, `"one-time"`

- Keyword gefunden **und kein OTP mitgeschickt** → `throw APIError.otpRequired`
- Keyword gefunden **aber OTP bereits mitgeschickt** → `throw APIError.unauthorized` (falscher Code)
- Kein Keyword → `throw APIError.unauthorized` (wie bisher)

### `checkConnection` entfällt komplett

`checkConnection` (Basic Auth gegen `/api/tags/`) wird **nicht mehr aufgerufen**. Bei aktivierter 2FA blockiert der Server Basic Auth mit 401 — der Flow käme nie bis `fetchToken`. `fetchToken` übernimmt alle Validierungen (OTP-Erkennung, falsche Credentials, Netzwerkfehler).

## UI-Änderungen (`LoginView.swift`)

### Neue States

```swift
@State private var otpRequired = false
@State private var otpCode = ""
```

### OTP-Feld (konditionell)

```swift
if otpRequired {
    VStack(spacing: 6) {
        Text("Gib deinen Authentifikator-Code ein")
            .font(.caption).foregroundColor(.secondary)
        TextField("6-stelliger Code", text: $otpCode)
            .textFieldStyle(.roundedBorder)
            .keyboardType(.numberPad)
            .multilineTextAlignment(.center)
    }
}
```

### Button-Label

```swift
Button(otpRequired ? "Code bestätigen" : "Login") { login() }
```

### Login-Funktion

```swift
private func login() {
    isChecking = true
    errorMessage = ""
    Task {
        do {
            if !otpRequired {
                try await PaperlessAPI.checkConnection(...)
            }
            let token = try await PaperlessAPI.fetchToken(
                serverUrl: store.serverUrl,
                username: store.username,
                password: password,
                otp: otpRequired ? otpCode : nil
            )
            KeychainService.saveToken(token.trimmingCharacters(in: .whitespacesAndNewlines), for: store.serverUrl)
            store.isDemoMode = false
            onConnect()
        } catch APIError.otpRequired {
            otpRequired = true
            errorMessage = ""
        } catch APIError.unauthorized {
            errorMessage = otpRequired ? "Falscher Code" : "Login fehlgeschlagen"
        } catch {
            errorMessage = error.localizedDescription
        }
        isChecking = false
    }
}
```

## Fehlerbehandlung

| Szenario | Verhalten |
|----------|-----------|
| Kein 2FA, korrektes Passwort | Normaler Login, kein OTP-Feld |
| 2FA aktiv, kein OTP im Request | OTP-Feld erscheint, kein Fehlertext |
| 2FA aktiv, falscher Code | „Falscher Code" im Fehlertext, OTP-Feld bleibt sichtbar |
| 2FA aktiv, korrekter Code | Login erfolgreich |
| Server gibt 400 ohne OTP-Keyword | „Login fehlgeschlagen" (wie bisher) |
| Basic Auth bei aktivierter 2FA | `checkConnection` entfernt — kein Vorcheck mehr |

---

## Nicht im Scope

- TOTP-Code direkt in der App generieren (kein Authenticator-Ersatz)
- Backup-Codes
- Hardware-Keys (WebAuthn/FIDO2)
- Einmaliges „Gerät merken"
